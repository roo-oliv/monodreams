# Plan: Fix Crash on Blender Level Load

## Context
After implementing the `-collider` child import feature, the game crashes when loading the Blender level. Two errors appear:
1. `LevelLoadRequestSystem` tries to load `"Blender_Level"` as an LDtk file and fails (pre-existing, caught)
2. `SpritePrepSystem.CreateNinePatchTexture` crashes with "Texture height must be greater than zero"

## Root Cause Analysis

**The crash chain:**
1. `ColliderDebugSystem` is instantiated at `LoadLevelExampleGameScreen.cs:284` even though the `debugSystems` variable is commented out of the pipeline at line 296
2. Its constructor (`ColliderDebugSystem.cs:33-34`) subscribes to `SubscribeEntityComponentAdded<BoxCollider>` and `SubscribeEntityComponentAdded<ConvexCollider>` — these subscriptions are **always active**
3. When `BlenderLevelParserSystem.ApplyColliderChild` sets a `ConvexCollider` on a parent entity, the subscription fires and `AddDebugLinesToConvexEntity` creates a debug entity with `NinePatchData` and `Size = aabb.Size`
4. The `elephant-kid-collider` has **degenerate vertices** — all 32 vertices have `Y = 0.0` (the mesh is on the XY plane in Blender instead of the XZ plane). This means AABB height = 0
5. `CullingSystem` adds `Visible` to the debug entity (it's in camera bounds)
6. `SpritePrepSystem` tries to create a `RenderTarget2D` with height 0 → crash

**The degenerate vertices issue:** The `elephant-kid-collider` mesh in Blender lies flat on the XY plane. Our exporter maps `(X, -Z)` to game coords, so Z=0 for all vertices → all game Y = 0. The mesh needs to be on the XZ plane for proper 2D projection (this is a Blender authoring issue, but the code should handle it gracefully).

## Fixes

### 1. Guard ColliderDebugSystem subscription
**File: `MonoDreams.Examples/Screens/LoadLevelExampleGameScreen.cs`**

The `ColliderDebugSystem` is constructed but never used. Since its constructor subscribes to world events, it creates ghost debug entities. Wrap the construction in a conditional or move it behind the same comment gate as `debugSystems` usage:

```csharp
// Only construct debug systems when they'll actually be used
// var debugSystems = new ParallelSystem<GameState>(_parallelRunner,
//     new ColliderDebugSystem(_world, _graphicsDevice)
// );
```

Comment out the `new ColliderDebugSystem(...)` call itself, not just its inclusion in the pipeline.

### 2. Add degenerate-vertex guard in ApplyColliderChild
**File: `MonoDreams/System/Level/BlenderLevelParserSystem.cs`**

After building `modelVertices` from the JSON data, compute the AABB and check that both width and height are > 0. If degenerate (all collinear or all coincident), log a warning and skip applying the collider.

```csharp
// Check for degenerate shape (all vertices collinear)
float minX = float.MaxValue, maxX = float.MinValue;
float minY = float.MaxValue, maxY = float.MinValue;
foreach (var v in modelVertices) { ... }
if (maxX - minX < 0.01f || maxY - minY < 0.01f)
{
    Logger.Warning($"Collider child '{colliderObj.Name}': degenerate shape (collinear vertices), skipping.");
    return;
}
```

### 3. Defensive guard in ColliderDebugSystem (defense-in-depth)
**File: `MonoDreams/System/Debug/ColliderDebugSystem.cs`**

In `AddDebugLinesToConvexEntity`, check that `aabb.Size` has both dimensions > 0 before creating the debug entity. Same check in `AddDebugLinesToEntity` for `boxCollider.Bounds.Dimension()`.
